I want to create an AI/ML model. The model will ingest pairs of .WAV audio files and .JSON config files during training. The model will ingest .WAV audio files during inference. The purpose of the model is predict the synthesizer filter type and frequency parameter values which were used to generate the .WAV file during inference. During training, the model will learn the relationships between the .WAV audio and the filter type and frequency parameter values by reading those two values from the .JSON config file with a name corresponding to the matching .WAV audio file.

For the sake of this MVP experimental development project, all of the .WAV files will be of the same size, and the audio will always be in the same format of 32-bit floating-point 48 kHz stereo, and will always have the same length of 1.1 seconds.

I have at least 300_000 pairs of WAV and JSON files in my "raw" dataset, and that number will probably be much larger in the future.

There are 108 filter type values, which are strings like ['FPhs 12HL6+',
 'FPhs 12HL6-',

 'Low EQ 12',
 'Band EQ 12',
 'Ring Mod',

 'SampHold-',
 'Combs',
 'Reverb',
 'Formant-I',
 'Formant-II',
 'Formant-III',
 'Bandreject',
 'Dist.Comb 1 LP'] and so on.

The filter frequency values can be anything from nearly 0 Hz to 24 Hz. For this initial MVP, I am only using the Serum 2 synthesizer. This synthesizer's filter frequency values are limited to a range of 8 Hz to 22050 Hz.

I believe that this program may benefit from using two separate models:
* one model to learn and predict frequency values, because these values lie along a continuum
* a second model to learn and predict filter type values, because these values are a finite set of categories and not a continuum.

I believe that the optimal approach is probably to encode the .WAV files' audio data into Mel spectrograms, and/or to use a multi-resolution short-term Fourier transform; encode the filter frequency values as either their raw values in Hz, or normalize them to a range such as [0, 1]; and to encode the filter type values in whatever way you deem to be optimal. Consider the pros and cons of transforming the raw audio data into multiple representations, such as using both a Mel-Spectrogram representation (to more closely model human perception of the relative loudness of different frequency bands) and a complex spectrogram (to capture stereo imaging and phase information, which may be relevant when using a phasing or flanging filter type); or using both a "stock" complex spectrogram and an "augmented" complex spectrogram based on the results of applying a perceptual weighting algorithm to the original "raw" audio data.

In order to handle stereo .WAV audio data, as well as account for stereo imaging and phase differences created by some filter types, I think that we should use something like `spectrogram_real_L, spectrogram_imag_L, spectrogram_real_R, spectrogram_imag_R`.

The filter type predictions must be very accurate, since these values do not represent a continuum, but totally separate categories.

The filter frequency value predictions do not need to be extremely precise, especially in the highest frequency ranges, as the human ear is unlikely to be able to discern a difference between, e.g., a low-pass filter with a frequency parameter value of 15000 Hz versus 13500 Hz or 16000 HZ. However, the human ear can be more sensitive to the human vocal range, as well as extremely low-frequency differences, so it is important that a filter frequency value of 60 Hz is not approximated by 25 Hz or 180 Hz (though something in the range of about 50 to 75 Hz would probably be acceptable as an approximation to the ground truth of 60 Hz).
